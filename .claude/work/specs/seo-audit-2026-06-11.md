# SEO Audit — minicuration.com

**Date:** 2026-06-11
**Scope:** Full technical + on-page audit of repo source at `/home/user/minicuration` (deployed source of truth for https://minicuration.com on Vercel). Live-site fetches were not possible from this environment — the site returns HTTP 403 to non-browser user agents, which is itself a finding (C-1). All evidence below is from source files; pages audited: `index.html`, `shop.html`, all 6 `shop/*.html`, `about.html`, `journal.html`, all 3 `journal/*.html`, `artists/index.html`, `artists/jfeelgood.html`, `policies.html`, `builder.html`, `404.html`, `sitemap.xml`, `robots.txt`, `llms.txt`, `vercel.json`, `api/`, `css/`, `js/`.

## Executive Summary

- **The single existential risk is the 403-to-bots behavior.** Repo-level SEO is unusually clean for a hand-built static site, but none of it matters if Googlebot receives a 403. This must be verified in Vercel's firewall settings before anything else.
- **All 16 JSON-LD blocks parse as valid JSON** and the sitemap exactly matches the set of indexable pages (16/16, with `builder.html` correctly noindexed and excluded). Titles, descriptions, and canonicals are present and unique on every indexable page. This is a strong baseline.
- **The biggest on-page risks are structured-data drift** (Product schema hardcodes `InStock` while inventory is dynamic) and **duplicate-URL hygiene** (`/index.html` vs `/`, `/artists/index.html` vs `/artists/`).
- **Quick wins:** fix the 404 page's relative links, fix a sitewide grammar bug on all 6 product pages, resolve a factual contradiction about ACEO's origin between `journal.html` and the ACEO article, and normalize product title patterns.

## Critical

### C-1. Live site returns 403 to non-browser user agents — verify search/AI crawler access immediately
- **Affected:** Entire site (Vercel project settings, not repo files)
- **Problem:** Automated fetches of https://minicuration.com receive HTTP 403 (observed by this and a sibling agent run). If this comes from Vercel's Attack Challenge Mode, Bot Protection, or a WAF custom rule that challenges all non-browser traffic, it can block Googlebot, Bingbot, and the AI crawlers (`GPTBot`, `ClaudeBot`, `PerplexityBot`, `CCBot`) that `robots.txt` explicitly *invites* (lines 4–20). A sustained 403 to Googlebot leads to deindexation. Nothing in the repo (`vercel.json` has only a www redirect, no headers/firewall config) causes this, so it is configured in the Vercel dashboard.
- **Impact:** High (potentially total loss of organic visibility)
- **Fix (testable):**
  1. In Vercel dashboard → Project → Firewall, check whether Attack Challenge Mode or Bot Protection is enabled; if so, set verified bots (search engine + AI crawler categories) to Allow, or scope challenges to non-verified traffic only.
  2. Verify with Google Search Console → URL Inspection → Live Test on `https://minicuration.com/` and `https://minicuration.com/shop/dreamfall.html`: both must return "URL is available to Google."
  3. Verify `curl -A "Mozilla/5.0 (compatible; Googlebot/2.1; +http://www.google.com/bot.html)" -I https://minicuration.com/` returns 200 (Vercel verified-bot allowlisting is by IP+UA, so also rely on the GSC live test as ground truth).
  4. Confirm GSC Crawl Stats show no spike in 403 responses over the following week.

### C-2. 404 page uses document-relative links — navigation breaks for any 404 under `/shop/`, `/journal/`, `/artists/`
- **Affected:** `404.html` (nav lines ~30–36, footer lines ~46–51)
- **Problem:** Nav and footer links are `href="shop.html"`, `href="about.html"`, etc. The 404 page is served at *whatever path* was requested, so a hit on `/shop/sold-out-design.html` or `/journal/old-post.html` renders a 404 whose links resolve to `/shop/shop.html`, `/journal/about.html`, etc. — all of which are themselves 404s. Users and crawlers landing on a dead deep URL get a dead-end page whose every recovery link is also dead (only the "Browse the Collection" button and footer logo use root-relative `/shop.html` / `/index.html`).
- **Impact:** High (soft-404 chains, crawl waste, lost users at exactly the moment recovery matters)
- **Fix (testable):** Change all nav/footer hrefs in `404.html` to root-relative (`/shop.html`, `/about.html`, `/journal.html`, `/policies.html`). Test: request `https://minicuration.com/shop/nonexistent.html`, click every nav/footer link on the rendered 404 — each must return 200.

### C-3. Product schema hardcodes `"availability": "InStock"` while real inventory is dynamic
- **Affected:** All 6 `shop/*.html` (JSON-LD `Offer` block); inventory served live by `api/stock.js`
- **Problem:** All six Offers declare `https://schema.org/InStock` (verified by grep). The visible page, however, flips to "Sold out" client-side from `/api/stock`. When edition 50 sells, Google continues to read `InStock` from the static HTML indefinitely — a structured-data/content mismatch that can trigger manual-review issues in Search Console, misleading rich results, and a bad click experience ("In stock · $23" in SERP → sold-out page). For a business whose entire premise is permanent sell-outs, this *will* happen for every product eventually.
- **Impact:** High (rich-result integrity for the site's only revenue pages)
- **Fix (testable):** Add a step to the sell-out workflow (the same event that flips `/api/stock`): update the product page's JSON-LD to `"availability":"https://schema.org/SoldOut"` and redeploy. Alternatively, generate availability at build time from the inventory table. Test: for any sold-out slug, `curl` the page source and assert the Offer block contains `SoldOut`; validate via Google Rich Results Test.

## Important

### I-1. Duplicate URL variants: `/index.html` vs `/` and `/artists/index.html` vs `/artists/`
- **Affected:** `artists/index.html` (canonical, line 12), `sitemap.xml` (line 29), every page's nav logo (`href="index.html"` / `../index.html`), JSON-LD breadcrumbs on `artists/jfeelgood.html`
- **Problem:** Vercel serves both directory and `index.html` forms with 200. The homepage handles this correctly (internal links go to `index.html` but canonical points to `/`). The artists section does it backwards: canonical and sitemap both use the ugly `/artists/index.html` form, splitting the page's identity and shipping a non-clean URL to search results. Internal nav links to `index.html` also leak the `/index.html` duplicate into the crawl graph (mitigated by canonical, but unnecessary).
- **Impact:** Medium
- **Fix (testable):**
  1. Change canonical in `artists/index.html` to `https://minicuration.com/artists/`; update `sitemap.xml` line 29 and the BreadcrumbList item in `artists/jfeelgood.html` to match.
  2. Change all nav/footer logo links from `index.html`/`../index.html` to `/`.
  3. Optionally add a redirect in `vercel.json` from `/index.html` → `/` and `/artists/index.html` → `/artists/` (308/301). Test: both URLs resolve, canonical form is the one in sitemap, GSC reports one indexed URL per page.

### I-2. `artists/index.html` is a near-orphan and the Artist nav item is inconsistent across the site
- **Affected:** All page navs; `artists/index.html`
- **Problem:** The "Artist" nav link exists only on `about.html` (line 66) and `journal/what-is-aceo.html` (line 101) — no other page's nav includes it. `artists/index.html`'s only inbound HTML link in the entire site is the breadcrumb on `artists/jfeelgood.html`. The artists hub is effectively orphaned (sitemap-only discovery), and `artists/jfeelgood.html` — a strong E-E-A-T asset — is reachable only via product-page byline links, two journal in-body links, and the inconsistent nav.
- **Impact:** Medium (link equity, E-E-A-T surfacing, crawl depth)
- **Fix (testable):** Add "Artist" (→ `/artists/jfeelgood.html` or `/artists/`) to the nav and/or footer on **all** pages, identically. Test: every indexable page contains at least one `<a>` to the artists section; `artists/index.html` has ≥ 2 inbound internal links from non-artists pages.

### I-3. Product Offer schema missing `shippingDetails` and `hasMerchantReturnPolicy`
- **Affected:** All 6 `shop/*.html` JSON-LD
- **Problem:** Google's product rich results increasingly surface shipping and return info from `OfferShippingDetails` and `MerchantReturnPolicy`. The site has clean, simple facts to declare ($5 flat US shipping — stated on-page; 30-day damaged-item returns — stated in FAQ and `policies.html`) but doesn't declare them. Competitors with complete Offer markup get richer listings.
- **Impact:** Medium
- **Fix (testable):** Add to each Offer:
  `"shippingDetails":{"@type":"OfferShippingDetails","shippingRate":{"@type":"MonetaryAmount","value":"5.00","currency":"USD"},"shippingDestination":{"@type":"DefinedRegion","addressCountry":"US"}}` and `"hasMerchantReturnPolicy":{"@type":"MerchantReturnPolicy","applicableCountry":"US","returnPolicyCategory":"https://schema.org/MerchantReturnFiniteReturnWindow","merchantReturnDays":30,"returnMethod":"https://schema.org/ReturnByMail","returnFees":"https://schema.org/FreeReturn"}` (align values with the actual policy in `policies.html` before shipping). Test: Rich Results Test shows shipping and returns sub-features detected with zero errors on all 6 pages.

### I-4. Factual contradiction about ACEO's origin between journal index and the article itself
- **Affected:** `journal.html` (line 78) vs `journal/what-is-aceo.html` (line 118); `index.html` journal teaser is vague but adjacent
- **Problem:** `journal.html` says the format started "when a collector named **Sandy Ronaldson** started listing miniature artworks"; the article it links to says it was "artist **Lisa Luree**." One of these is wrong, both are indexed, and the topic is the site's primary informational keyword ("what is ACEO"). Contradicting yourself on your own pillar topic damages E-E-A-T and AI-answer-engine citability. (The commonly cited history: ATCs predate eBay; the ACEO category/term is generally credited to eBay seller Lisa Luree, "bone*diva," circa 2004.)
- **Impact:** Medium (content trust on the highest-value informational query)
- **Fix (testable):** Verify the history, pick the correct attribution, and make `journal.html`, `index.html` teaser, and `journal/what-is-aceo.html` agree verbatim. Test: `grep -ri "ronaldson\|luree" *.html journal/*.html` returns a single consistent name.

### I-5. Sitewide grammar error on all six product pages
- **Affected:** All 6 `shop/*.html`, "The Original Painting" section (e.g., `shop/dreamfall.html` line 101)
- **Problem:** Template bug: "It **currently in** artist's studio and is not for sale" (missing "is") appears on all six product pages (verified by grep). Money pages with template-level grammar errors read as low-quality to users and quality systems alike.
- **Impact:** Medium (content quality on all revenue pages)
- **Fix (testable):** Change to "It is currently in the artist's studio and is not for sale." on all six pages. Test: `grep -r "It currently in" shop/` returns nothing.

### I-6. Product title tags: two inconsistent patterns, and the longer one truncates in SERPs
- **Affected:** All 6 `shop/*.html` titles; homepage title
- **Problem:** Three products use "…— Limited Edition Mini Art Print | Minicuration" (~71 chars — truncates at ~60) and three use "…— ACEO Print | Minicuration" (~54 chars — fits). The homepage title is 68 chars ("Minicuration — ACEO Limited Edition Art Prints | Trading Card Sized") and will truncate after "…Prints". Inconsistent patterns also dilute the template's keyword targeting (is the product keyword "mini art print" or "ACEO print"?).
- **Impact:** Medium
- **Fix (testable):** Standardize all six to one ≤60-char pattern, e.g. `[Name] by JFeelgood — ACEO Art Print | Minicuration`. Shorten homepage to ≤60 chars, e.g. `ACEO Limited Edition Art Prints | Minicuration`. Test: every `<title>` ≤ 60 characters; one pattern across all products; `og:title`/`twitter:title` match their page's `<title>` (currently mismatched on `shop.html` and `shop/sweet-dreams.html`).

### I-7. `Article` schema on `what-is-aceo.html` is missing `author`; breadcrumb is nested in the wrong type on all 3 articles
- **Affected:** `journal/what-is-aceo.html` (no `author` — the other two articles have it, verified by grep); all 3 `journal/*.html` (BreadcrumbList nested as a `breadcrumb` property *of the Article*)
- **Problem:** `author` is a key field for Article rich results and AI citation; the flagship article omits it. Separately, `breadcrumb` is a property of `WebPage`, not `Article` — nested this way Google may ignore the BreadcrumbList entirely (product pages do it correctly as a separate `@graph` node).
- **Impact:** Medium
- **Fix (testable):** Add `"author":{"@type":"Person","name":"JFeelgood","url":"https://minicuration.com/artists/jfeelgood.html"}` (or the actual author) to `what-is-aceo.html`; move BreadcrumbList out to a sibling node in `@graph` on all three articles. Test: Rich Results Test detects both Article and Breadcrumb features with no warnings on all 3 URLs.

### I-8. Product pages never link to the Journal — one-way silo
- **Affected:** All 6 `shop/*.html`
- **Problem:** Journal articles link generously to products and shop (verified: `how-paintings-become-prints.html` links all 6 product pages), but no product page links to any journal article (grep for `journal/` in `shop/*.html` returns nothing). The hub-and-spoke loop is open: product pages hoard equity and pass none to the informational cluster, and buyers never see the trust-building content ("What is ACEO?" is the exact question a first-time buyer has).
- **Impact:** Medium
- **Fix (testable):** Add one contextual sentence to the product template, e.g. in the Edition Details or FAQ section: `New to the format? Read <a href="../journal/what-is-aceo.html">What Is ACEO?</a>` and link "How paintings survive card scale" from the "Why This Works at Card Scale" section. Test: each product page contains ≥ 1 link to `journal/`.

### I-9. Organization `logo` is a product photo, and `og:image` is the same product photo on 6 non-product pages
- **Affected:** `index.html` JSON-LD (line 25: `"logo":"https://minicuration.com/image/sweetdreams-case1.webp"`); `og:image` on home, about, journal index, all 3 journal articles, both artist pages
- **Problem:** The Organization logo should be an actual brand logo (square, ≥112×112) — Google may use it in knowledge panels and brand SERPs; a product case photo will look wrong or be rejected. Reusing `sweetdreams-case1.webp` as the share image for every informational page makes journal articles indistinguishable when shared and wastes the image's relevance signal.
- **Impact:** Medium-Low
- **Fix (testable):** Create a real logo file (e.g., `/image/minicuration-logo.png`, square PNG ≥512×512), reference it in Organization schema; give each journal article a topical `og:image`. Test: Rich Results Test shows valid Organization logo; each journal article's `og:image` URL is unique.

## Nice-to-have

### N-1. Inconsistent font loading: `shop.html` + product pages use render-blocking Google Fonts while the rest self-host
- **Affected:** `shop.html` (line 32), all 6 `shop/*.html`, `builder.html`
- **Problem:** `css/base.css` self-hosts woff2 with `font-display:swap` (and the files exist in `/fonts/`), but the shop and product templates instead load `fonts.googleapis.com` CSS — an extra render-blocking third-party request chain on the money pages, hurting LCP exactly where conversion happens. `shop.html` also carries ~150 lines of inline CSS duplicating `base.css` instead of linking it.
- **Fix:** Switch shop/product templates to `/css/base.css` fonts (drop the Google Fonts `<link>`s and preconnects). Test: no requests to `fonts.googleapis.com`/`fonts.gstatic.com` on any indexable page; Lighthouse LCP on a product page ≤ previous baseline.

### N-2. Unused image preload on the homepage
- **Affected:** `index.html` line 27 (`<link rel="preload" as="image" href="image/spin/dreamfall_artwork.webp">`)
- **Problem:** No element on the homepage references `dreamfall_artwork.webp` (the hero is text-only; spotlight images are `asimplemeditation`/`skymiles`). The preload downloads ~unused bytes on every visit and logs a browser warning. On `shop.html` the same preload targets an image that is also marked `loading="lazy"` — contradictory hints.
- **Fix:** Remove the preload from `index.html` (or point it at the first spotlight image and remove that image's `loading="lazy"`); on `shop.html`, drop `loading="lazy"` from the first above-the-fold card or remove the preload. Test: Chrome DevTools console shows no "preloaded but not used" warning.

### N-3. `robots.txt`: add `Disallow: /api/`
- **Affected:** `robots.txt`, `api/stock.js`
- **Problem:** `/api/stock` returns JSON 200 to GET and is crawlable. Harmless but pointless crawl; could get indexed as a naked JSON page.
- **Fix:** Add `User-agent: *` → `Disallow: /api/` (keep everything else `Allow`). Test: `robots.txt` validator shows `/api/stock` blocked, all HTML pages allowed.

### N-4. Sitemap `lastmod` is bulk-set to the same date for all 16 URLs
- **Affected:** `sitemap.xml` (every `<lastmod>` is `2026-06-10`)
- **Problem:** Uniform lastmod across all pages signals "auto-stamped, not real," and Google then ignores the field. The articles say "Published May 2026"; their schema says `datePublished: 2026-05-16/17`.
- **Fix:** Set lastmod per-page to actual last content change (match `dateModified` in each page's schema); only bump on real edits. Test: `lastmod` values vary and match on-page `dateModified`.

### N-5. `/shop/` and `/journal/` directory URLs 404
- **Affected:** Vercel routing (no `shop/index.html` or `journal/index.html`; `vercel.json` has no redirects for these)
- **Problem:** Users (and some link-builders) truncate URLs to the directory. `/shop/` and `/journal/` currently 404 even though `/shop.html` and `/journal.html` exist — wasted inbound equity if anyone links the directory form.
- **Fix:** Add redirects in `vercel.json`: `/shop/` → `/shop.html`, `/journal/` → `/journal.html` (or migrate to `cleanUrls: true` with a full redirect map as a separate, planned project — do not do this casually; it changes every canonical URL). Test: `curl -I https://minicuration.com/shop/` returns 301/308 to a 200 page.

### N-6. FAQPage schema duplicated verbatim across all 6 product pages, and FAQ rich results are restricted anyway
- **Affected:** All 6 `shop/*.html` (identical 4-question FAQPage), `about.html`, `journal/what-is-aceo.html`
- **Problem:** Since Aug 2023 Google shows FAQ rich results almost exclusively for authoritative gov/health sites, so these blocks won't produce SERP features; six identical FAQPage blocks also declare the same questions as the "main entity" of six different URLs. Not harmful, but noise. (Keep the *visible* FAQ content — it's good for users and AI extraction.)
- **Fix:** Optional: remove FAQPage JSON-LD from product pages (keep on `about.html` and `what-is-aceo.html` where questions are page-specific), or differentiate per-product questions. Test: Rich Results Test still passes Product/Breadcrumb on all 6.

### N-7. Meta description polish
- **Affected:** `about.html` (~187 chars, will truncate), `shop/sky-miles.html` (~171 chars), lowercase medium names in `shop/dream-mountain.html`, `shop/sweet-dreams.html`, `shop/veritas.html` descriptions ("acrylic on canvas.")
- **Fix:** Trim to ≤160 chars; sentence-case the medium. Test: all descriptions 70–160 chars, no lowercase sentence starts.

### N-8. `artists/index.html` missing `twitter:*` tags; artist-page portfolio cards have generic sub-text
- **Affected:** `artists/index.html` (og only, no twitter card tags); `artists/jfeelgood.html` portfolio cards all say "Trading Card Sized Art Prints"
- **Fix:** Add the standard twitter card block; replace card sub-text with per-work medium (e.g., "Acrylic on Canvas · $23"). Test: tags present; sub-text unique per card.

### N-9. Homepage h1 carries no category keyword
- **Affected:** `index.html` (h1: "Limited Edition. Small Format, Real Art.")
- **Problem:** The h1 is brand-voice only; the keyword load ("ACEO", "art prints", "trading card") sits in the eyebrow `<p>` and title tag. Acceptable, but the single highest-weight on-page element is keyword-empty.
- **Fix (only if it survives design review):** Work one concrete noun into the h1, e.g. "Limited Edition Art Prints. Trading Card Sized." Test: h1 contains "art print" or "ACEO"; passes frontend-design review.

## What's already good (no action)

- All 16 indexable pages have unique titles, unique meta descriptions, self-referencing canonicals, `index, follow`, exactly one `<h1>` (builder.html's 2 h1s don't matter — it's `noindex, nofollow`), `lang="en"`, and responsive viewport tags.
- **Sitemap ⇄ filesystem parity is exact:** 16 URLs = 16 indexable pages; `builder.html` and `404.html` correctly excluded; image sitemap extensions used correctly on product entries.
- All JSON-LD blocks parse (validated programmatically); Product + VisualArtwork + BreadcrumbList graph on product pages is well-constructed; `priceValidUntil` present.
- `robots.txt` allows all major search and AI crawlers explicitly and declares the sitemap; `llms.txt` is accurate and matches the live catalog.
- www → apex 301 in `vercel.json`; descriptive keyword-bearing alt text on essentially every image; descriptive hyphenated URLs; visible breadcrumbs with `aria-label` on deep pages; journal articles link products, shop, artist, and each other; `prefers-reduced-motion` respected.
- 404 page returns correct semantics (noindex, on-brand recovery CTA) — once C-2's link bug is fixed.

## Overall Health Summary

Repo-level SEO is in the top decile for a small hand-built static site: complete metadata, valid structured data, exact sitemap parity, deliberate robots/llms files, and a real informational content cluster targeting the right keyword ("ACEO"). The risks are concentrated off-page-source: a firewall that may be 403-ing the crawlers the site is otherwise courting, structured data that will silently go stale the moment the business succeeds (sell-outs), and a handful of template bugs (404 links, grammar, title patterns) that are cheap to fix. Site architecture is sound except the artists section, which is under-linked.

**Top 3 highest-impact fixes:**
1. **C-1 — Verify and fix the 403-to-bots behavior in Vercel firewall settings**, then confirm with a GSC live URL inspection. Everything else is moot until this is confirmed resolved.
2. **C-3 — Wire Product schema `availability` into the sell-out workflow** so sold-out editions serve `SoldOut`, protecting rich-result integrity on every revenue page.
3. **C-2 + I-5 — Fix the 404 page's relative links and the "It currently in" template typo** — two trivial template edits that remove a crawler dead-end and a sitewide quality flaw on all six money pages.
